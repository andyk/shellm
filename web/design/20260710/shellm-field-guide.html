▶ shellm field guide

Everything is a line in one file

A shellm identity — its thoughts, its actions, the shell commands it runs, your chat messages to it — is an append-only JSONL file. This page is a map of that file: the record format, who writes which lines, and how the pieces trigger each other.

one step · one line of trajectory.jsonl
{ "step_id": "7c02…", "ts": "2026-07-09T16:02:11.483-0700", "type": "thought", "source": "inner_monologue", "content": "nick asked about disk usage — the actor is on it" }
   └─ mandatory: step_id · ts · type ──┘  └─ everything else is optional and depends on the type ─┘
01 · The record format

One shape, tagged by type

There is no schema zoo. Every step is a JSON object with three mandatory fields — step_id (UUID), ts (timestamp), type — plus whatever payload fields that type carries. traj append validates nothing beyond “valid JSON with a type.” It's a tagged union over one open record, the standard event-log pattern.

The tag carries meaning the fields can't: thought, idle, and action have byte-identical field sets (content + source), but an action makes the dispatcher fire the actor, a thought is broadcast for other thinkers, and an idle is a heartbeat the viewer folds away. The type is intent.

Trajectory vs. mind log: a trajectory is any such file — the record of any shellm run, living at <hex8>-<slug>/trajectory.jsonl with a sibling blobs/ for oversized output. A mind log is the special role one trajectory plays: an identity's root trajectory (pointed at by info.txt), used as a shared bus. Every mind log is a trajectory; most trajectories aren't mind logs.

Concurrent writers are safe because every write goes through traj append, which serializes appends with a mkdir-based lock (atomic, works on macOS and across Docker mounts). It guarantees no torn lines — and nothing more. Log order is just lock-acquisition order.

03 · Machinery vs. sourced steps

The fingerprint rule

“Machinery” is not a property of what caused a step — it's the fingerprint of who wrote the line. The shellm loop writes its seven types and never stamps source; thinker scripts always stamp theirs. One actor activation produces both kinds in the same log.

has source → a mind speaking

Written by a thinker script (or chat) directly onto the bus. source names the voice: inner_monologue, actor, chat, seed.

no source → the loop working

Written by a running shellm process: the run header, each iteration's reasoning + shell output, the final answer.

Since 2026-07-05 the actor runs shellm --traj <root>, pouring its machinery inline into the mind log (“flat era”) so other thinkers can see work in-flight — instead of hiding it in a forked child trajectory (“fork era”). The cost: machinery interleaves with concurrent monologue, and nothing ties a step to its run. Toggle below to see what the viewer has to reconstruct:

Note the monologue keeps thinking during the run — that interleaving is real concurrency, not noise.

04 · Three meanings of “fork”

Unix forks constantly; trajectories fork rarely

Easy place to get confused: executing bash every iteration is a unix fork, but it isn't a shellm fork. Three layers, only the last one creates structure in the log:

1 · unix process

Forks everywhere, invisible as structure. Each iteration fork/execs a bash subprocess (or docker exec); the dispatcher fork/execs thinker scripts. The only trace is captured stdout/stderr inside a shell-output step.

2 · log record

Who wrote the line — machinery vs. sourced (section 03). One iteration of the loop = a reasoning + shell-output pair, same trajectory, no forking of anything.

3 · trajectory topology

A fork step + child trajectory dir exist only when code inside a run invokes shellm again (without --traj). The child sees _SHELLM_PARENT_TRAJ_ID in its environment and writes the cross-link. --traj is precisely the “don't fork the topology” switch.

05 · Entry points

Three ways a loop starts — and one that isn't

Only three things ever run the shellm loop, and they're the same function with different trajectory plumbing:

you · imperatively 
shellm "mass of jupiter in kg"

Synchronous. Own fresh trajectory; answer on stdout.

a thinker · reactively 
shellm --traj "$TRAJ_ID" …

Fired by dispatch. Machinery inlined into the mind log; no fork step, no child.

generated code · recursively 
x=$(shellm "classify: $text")

Nested call from inside a run. Forks a child trajectory with cross-links.

chat repl is not on this list. It contains no LLM call at all: chat send is a single traj append of a message step, and the repl is a TUI wrapping append + tail. You-in-the-repl are architecturally a peer of inner_monologue — a sourced writer on the bus. A reply reaches you like this:

1

You type; a message step lands on the bus — chat send, from: nick, source: "chat". Nothing has “called an agent.”

2

The dispatcher matches the type — the actor subscribes to message; a guard skips messages not addressed to this identity (prevents reply-to-self loops).

3

The actor launches shellm --traj — the only loop in the story. Its prompt says: send any reply in your first bash block, then do follow-up work.

4

The reply is itself a bus step — generated code runs chat reply nick "…", another plain append. Your repl's tail renders it. Request/response is an illusion over publish/subscribe.

06 · The whole machine

One bus, many hands

identity shell picks the bus (env vars). thinkers start turns on the reactive side. Everything else writes or reads:

inner_monologuellm call → thought / idle / action
actorshellm --traj → machinery + replies
you · chatmessage steps
shellm loopreasoning / shell-output / final
↓ write↓ write↓ write↓ write
trajectory.jsonl — the mind log append-only · mkdir-lock serialized · order = arrival
↓ read↓ read↓ read↓ read
dispatchernew step → fire subscribed thinkers
chat repl / TUItail + render
web viewergroup runs, walk fork trees
shellm (resume)rebuild LLM context from steps

The loop that makes it feel alive: thinker steps trigger the dispatcher, the dispatcher fires thinkers, thinkers write more steps. Machinery steps land on the bus too, but today's thinker subscriptions tune them out — only sourced steps drive dispatch.

zimbot answers a question

A complete, simplified session with a made-up identity zimbot: two thinkers running, you in the chat repl. step_id/ts are dropped except where cross-links need them, and long content is trimmed — everything else is exactly what would land on disk.

# pick the bus: subshell with TRAJ_ID/ROOT_TRAJ_ID → zimbot's root trajectory
$ identity shell zimbot

# turn on the reactive side: dispatcher tails the mind log, fires these two on matching steps
$ thinkers start inner_monologue actor

# attach yourself as a bus client (append messages + tail the stream — no LLM in this command)
$ chat repl --from nick
